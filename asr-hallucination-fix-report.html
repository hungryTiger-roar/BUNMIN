1. beam_size=1 추가 — 속도

변경 1

왜 추가했는가

faster-whisper의 기본 beam_size는 5입니다. Beam search는 매 디코딩 스텝마다 상위 N개 후보를 동시에 추적하면서 최적 경로를 고르기 때문에, N이 클수록 정확도는 약간 올라가지만 디코딩 비용이 N배에 가깝게 늘어납니다.

beam_size=1은 매 스텝에서 가장 확률 높은 토큰 하나만 채택하는 greedy decoding입니다. 후보를 5개에서 1개로 줄이므로 디코딩이 그만큼 짧아집니다.

왜 정확도 손실을 감수해도 되는가
① NMT는 이미 같은 선택을 내렸음. backend/app/services/nmt_service.py의 CT2 / HF 양쪽 경로 모두 beam_size=1 / num_beams=1입니다. 이전 단계에서도 같은 trade-off로 속도를 택한 적 있음.
② 일반적으로 Whisper 계열에서 beam=1 vs beam=5의 WER 차이는 1~3% 수준이고, 한국어 단문/단발화 상황에서는 차이가 더 줄어드는 경향이 있음.
③ 2초 latency 예산이 절대 기준이고, ASR이 그 예산의 가장 큰 부분을 차지합니다 (측정값: 짧은 발화 ASR 0.4~1.1s).
기대 효과: ASR 디코딩 속도 약 20~30% 단축. 단발화 0.7~1.0s 케이스에서 100~300ms 절약 → 2초 예산에 여유 확보.
검증 필요: python evaluation/run_eval.py --model asr로 WER 회귀 폭이 1~3% 이내인지 확인. 5% 넘게 떨어지면 beam_size=2로 절충.
측정 결과 — 위 추론은 우리 환경에서 틀렸습니다 (2026-04-28)
실제 200개 한국어 강의 음성 샘플로 N=2 측정한 결과:
beam_sizeWER (deterministic)평균지연 (N=2)
5 (기본)46.2% ✓626ms (스프레드 81ms)
247.8%555ms (N=1)
1 (제안)48.1% ✗ +1.9%p591ms (스프레드 34ms)
왜 추론이 틀렸나:
  • 30% 속도 향상 가정 오류 — faster-whisper(CTranslate2)의 beam search 배칭 최적화가 GPU 병렬성을 거의 공짜로 흡수. 일반 PyTorch Whisper 벤치마크 수치를 우리 환경에 그대로 가져온 게 잘못.
  • 속도 차이 35ms는 노이즈 안 — 같은 설정 두 번 돌렸을 때 beam=5는 81ms, beam=1은 34ms 변동. 평균 차이가 자체 변동성에 묻힘.
  • 환각 방어 효과 역행 — beam search는 무음 구간에서 "시청해주셔서…" 같은 정형구 환각이 시작될 때 다른 경로로 빠져나갈 기회를 줌. greedy(beam=1)는 한 번 빨려들면 못 빠짐. 즉 "환각 차단"이 목표인 수정에서 환각 방어 메커니즘을 떼버린 자기모순.
현재 코드: beam_size=5로 영구 회복. 이번 보고서에서 살아남은 변경은 2~5번(필터 루프 + 임계값)뿐.

2. 단순 join → 필터 루프 — 환각 차단 인프라

변경 2

왜 코드 구조를 바꿨는가

수정 전 코드는 "".join(s.text for s in segments).strip()로 모든 세그먼트의 텍스트를 무조건 합쳤습니다. 이 구조에서는 환각으로 의심되는 세그먼트를 걸러낼 자리가 없습니다.

faster-whisper의 Segment 객체는 텍스트뿐만 아니라 다음 메타정보를 함께 줍니다:

  • seg.no_speech_prob — 이 세그먼트가 사실은 음성이 아닐 확률 (0~1)
  • seg.avg_logprob — 디코딩한 토큰들의 평균 log-probability (높을수록 모델이 확신, 보통 -0.5 ~ -0.1)
  • seg.compression_ratio — 텍스트 압축비, 너무 높으면 반복 환각

이 메타정보를 실제로 쓰려면 세그먼트를 한 번씩 들여다보는 루프가 반드시 필요합니다. 단순 join은 메타정보가 있어도 활용할 자리가 없는 구조였기 때문에, 필터 루프로 교체하는 것이 환각 차단의 전제 조건이었습니다.

왜 메타정보 기반 필터인가 — Whisper 환각의 정체
Whisper는 학습 데이터에서 "시청해주셔서 감사합니다", "다음 영상에서 만나요", "MBC 뉴스 ○○○입니다" 같은 정형구를 자주 봤습니다. 무음/잡음 구간을 디코딩할 때 모델은 "여기는 음성이 거의 없는데 뭔가 만들어내야 하니" 학습 데이터의 정형구를 출력합니다. 이때 no_speech_prob는 높고 avg_logprob은 낮습니다. 즉 모델 자신이 "이건 별로 자신 없다"고 신호를 보내는 것이고, 우리는 그 신호를 무시하지 않으면 됩니다.
구조적 의미: 이번 변경의 핵심은 임계값 자체가 아니라, "세그먼트 단위로 신뢰도를 보고 채택/기각을 결정하는 파이프라인 자리"를 코드에 만든 것입니다. 임계값은 추후 데이터 보면서 튜닝하면 되지만, 자리가 없으면 튜닝 자체가 불가능합니다.

3. 스킵된 세그먼트는 로그에 남김 — 튜닝 근거 수집

변경 3

왜 그냥 버리지 않고 print 했는가

if seg.no_speech_prob > 0.6: print(f"[ASR] 세그먼트 스킵 — no_speech_prob={seg.no_speech_prob:.2f}: {seg.text!r}") continue if seg.avg_logprob < -0.8: print(f"[ASR] 세그먼트 스킵 — avg_logprob={seg.avg_logprob:.2f}: {seg.text!r}") continue

스킵하는 게 목적인데도 굳이 텍스트와 점수를 같이 출력합니다. 이유는 단순합니다 — 임계값 0.6, -0.8이 맞는지 확신할 수 없기 때문입니다.

왜 로그가 결정적인가
임계값을 너무 보수적으로 잡으면 (0.6) → 환각이 통과합니다.
임계값을 너무 공격적으로 잡으면 (0.3) → 정상 발화도 스킵되어 자막이 비어 나옵니다.
이 트레이드오프의 정답은 우리 환경(마이크, 강의실 노이즈, 한국어 발화 패턴)의 실측 분포 없이는 결정 불가능합니다. 로그는 그 분포를 만드는 원시 데이터입니다.

로그를 어떻게 활용할 것인가

  • 거짓 양성(false positive) 발견 — 정상 발화인데 스킵된 케이스가 보이면 (예: 짧은 한국어 종결어미 "네.", "음."), 임계값을 완화.
  • 거짓 음성(false negative) 발견 — 환각이 통과한 경우 그 텍스트의 메타값을 직접 확인 → 임계값을 강화.
  • 마이크/환경별 분포 차이 확인 — 강의자별로 no_speech_prob 평균이 크게 다르면 환경 보정 필요.
다음 단계: asr_eos_design.md에서 결정한 "로깅 인프라 먼저, 임계값 튜닝 나중" 원칙과 같은 맥락. 현재는 print지만, 이후 구조화 로그(seg_skipped 이벤트)로 승격하면 분포 분석이 쉬워집니다.

4. no_speech_prob > 0.6 스킵 — 무음/잡음 환각 차단

변경 4

이 임계값이 의미하는 것

no_speech_prob는 Whisper 모델이 "이 세그먼트는 음성이 아닐 가능성"을 직접 예측한 값입니다 (0~1, 높을수록 비음성).

no_speech_prob해석현재 정책
0.0 ~ 0.3명확한 음성채택
0.3 ~ 0.6회색지대 — 음성일 가능성 더 큼채택 (보수적 선택)
0.6 ~ 1.0비음성 가능성 우세스킵
왜 0.6인가 — 보수적 시작값
Whisper 환각 차단 가이드들에서 흔히 권장되는 값은 0.4~0.6 범위입니다. 우리는 그 범위의 가장 안전한 끝(0.6)에서 시작했습니다.

이유: 한국어 종결어미("~습니다", "~네요")는 발화 말미에 에너지가 급격히 떨어지면서 마지막 짧은 세그먼트의 no_speech_prob가 의외로 높게 측정될 수 있습니다. 임계값을 0.4로 낮추면 정상 발화의 종결어미가 잘릴 위험이 있습니다.

이는 asr_eos_design.md가 EOS 판단에 RMS 단독 고정 임계값을 기각한 것과 같은 이유입니다 — 한국어 저에너지 종결어미는 잘리기 쉽습니다.
이 값은 잠정값입니다. 로그에서 다음 두 패턴을 보고 조정합니다:
• 환각이 통과 (예: "시청해주셔서 감사합니다") → 0.5나 0.4로 낮춤
• 정상 발화 종결어미가 스킵됨 → 0.7로 올림 또는 임계값 외에 다른 신호(텍스트 패턴 매칭) 추가

5. avg_logprob < -0.8 스킵 — 모델 확신도 극히 낮을 때만

변경 5

이 임계값이 의미하는 것

avg_logprob는 디코딩한 모든 토큰의 log-probability 평균입니다. log 값이라 음수가 정상이고, 0에 가까울수록 모델이 확신, -∞로 갈수록 자신 없음을 의미합니다.

avg_logprob해석현재 정책
-0.1 ~ -0.3매우 자신 있음 (정상 발화의 일반적 범위)채택
-0.3 ~ -0.6보통 (어려운 발화, 약한 잡음)채택
-0.6 ~ -0.8낮은 신뢰 — 회색지대채택 (보수적 선택)
< -0.8모델조차 거의 추측 수준스킵
왜 -0.8인가 — "모델 확신도가 극히 낮을 때만"
Whisper 공식 권장 컷오프는 -1.0이고, 환각 차단 가이드들은 -0.5 ~ -1.0 범위를 권장합니다. 우리는 그 범위의 보수적 끝쪽(-0.8)에서 시작했습니다.

이유: avg_logprob은 발화 난이도(빠른 말, 사투리, 전문용어, 영어 섞임)에 민감합니다. 강의 자료에는 영어 전문용어("Transformer", "Attention" 등)가 자주 섞여 들어오는데, 한국어 ASR 모델 입장에서 이런 토큰의 logprob은 자연스럽게 낮게 잡힙니다. 컷오프를 -0.6으로 두면 전문용어가 들어간 정상 발화가 통째로 사라질 위험이 있습니다.

-0.8은 "발화 자체가 거의 알아들을 수 없을 정도"의 신호로 한정됩니다. no_speech_prob 필터를 통과한 것 중에서도 추가로 한 번 더 거름.

두 필터의 관계 — 보완 구조

  • no_speech_prob = "음성 자체인가?" 판단 → 무음/잡음 환각 잡음
  • avg_logprob = "음성이긴 한데 알아들었나?" 판단 → 음성 위에 만든 환각 잡음

두 신호가 다른 종류의 실패를 잡기 때문에 OR 조건(둘 중 하나만 걸려도 스킵)으로 묶여 있습니다.

이 값도 잠정값입니다.
• 영어 전문용어 섞인 정상 발화가 스킵됨 → -0.9 또는 -1.0으로 완화
• 무의미한 음절 출력("어어 그 그"이 통과) → -0.7로 강화

6. 종합 — 이번 수정의 5가지가 어떻게 맞물리는가

① 속도 한 줄 (beam_size=1) + ② 환각 차단을 위한 코드 자리(필터 루프) + ③ 데이터 수집(스킵 로그) + ④⑤ 잠정 임계값 두 개

핵심은 "임계값을 잘 골랐다"가 아니라 "임계값을 데이터로 튜닝할 수 있는 구조를 만들었다"는 점입니다. 0.6과 -0.8은 잘 골랐다기보다 보수적으로 시작한 값이고, 실제 사용 로그에서 거짓 양성/음성을 보면서 단계적으로 조정합니다.

  5. (별개) Step 8 빠른 연속 발화 번들링 문제 — VAD 측 분포 로그 수집 후 본질 해법 (asr_eos_design.md의 4상태 머신) 진행

eval 한계에도 결론이 흔들리지 않는 이유:
  • WER 우위는 deterministic — production 음성이 더 어렵다 해도 같은 모델 같은 입력에서 beam=5가 항상 같거나 우세. 1.9%p 차이가 줄어들 수 있어도 뒤집히진 않음.
  • 속도 우위(beam=1)는 이미 노이즈 안 — production GPU 경합이 추가되면 노이즈만 더 커지지 beam=1의 35ms 우위가 살아날 이유 없음.
  • 환각 방어 효과는 eval에서 측정조차 안 됨 — 환각은 음성 없는 구간에서 터지는데 eval 데이터는 음성으로 가득. production에서 beam=5의 환각 방어 효과는 eval 결과보다 더 클 가능성.

Production sanity check 절차 (다음 단계)

eval 한계를 보완하는 가벼운 production 검증 절차:

  1. 코드 변경 적용 확인 — backend/app/services/asr_service.py의 beam_size=5 확인
  2. dev 서버 기동 — npm run dev 후 강의자/수강자 두 화면 띄움
  3. 5~10분 강의 시뮬레이션 — 다음 시나리오 골고루 포함:
    • 긴 한 문장 발화
    • 짧은 단발 발화 ("네", "음" 등 종결어미)
    • 발화 사이 짧은 침묵 (0.5~1초)
    • 의도적인 긴 침묵 (3초+) — 환각 트리거 테스트
    • 전문용어 섞인 발화 (영어 단어 포함)
  4. 두 신호 모니터링:
    • 백엔드 로그의 [LATENCY] seq=… | ASR=… | NMT=… | TTS=… | 전체=… — 전체 2초 예산 안에 들어오는지
    • 수강자 화면 자막 — "시청해주셔서 감사합니다" 등 환각 정형구 출력 여부
  5. 판정:
    • 전체 latency 평균 2초 이내 + 환각 미발생 → beam=5 유지
    • 환각 여전 → 임계값 강화 (no_speech_prob 0.6 → 0.5) 또는 정형구 블랙리스트 추가
    • 2초 예산 자주 위반 → P99 분석 후 다른 단계(NMT/TTS) 최적화 검토

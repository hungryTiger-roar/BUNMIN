# ASR → MT 파이프라인 원인 분석

> 작성일: 2026-04-24 / 최종 수정: 2026-04-29  
> 목적: 파이프라인 개선 전 각 모델별 문제 원인을 확인된 것과 미확인된 것으로 분리하여 정리

---

## 분석 배경

ASR → MT → TTS 파이프라인에서 속도와 품질 두 측면에서 문제가 발생했다. (백엔드 TTS는 2026-04-29 제거됨 — 클라이언트 WASM으로 이전)  
한 사람이 전체 파이프라인을 수정하는 방식은 원인 파악과 효과 검증이 어렵기 때문에,  
다음 주부터 모델별 담당자를 지정하여 각자 원인을 깊이 파고드는 방식으로 전환한다.

---

## ASR 문제

### 속도

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 모델 자체 처리 시간 (Cohere 모델 크기) | **확인됨** | LATENCY 로그에서 ASR 구간 수치 측정 가능 |
| GPU 경합 (NMT와 GPU 동시 점유) | **✅ 적용 완료** | `_asr_semaphore` + `_nmt_semaphore` 분리 적용 — ASR GPU 직렬화, NMT CPU 독립 처리, TTS 세마포어 밖 분리 |
| 입력 오디오 길이와 처리 시간 비례 여부 | **미확인** | 짧은 발화(1~3s) vs 긴 발화(5s+) 구간별 비교 필요 |

### 품질 (환각)

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 노이즈 입력 (VAD 통과한 소음) | **확인됨** | 반복 문자, 중국어 혼입 패턴이 로그에 기록됨 |
| 짧고 조용한 발화 통과 | **확인됨** | `_validate_audio` 필터가 잡는 케이스 존재 |
| 모델 특성 (Cohere 한국어 환각 경향) | **미확인** | 동일 오디오로 다른 모델(Moonshine 등) 비교 실험 필요 |

### 미확인 항목 검증 방법

- LATENCY 로그를 발화 길이별(짧음/긺)로 분류하여 수십 건 수집
- 동일 오디오 샘플로 Cohere vs Moonshine v2 Korean 결과 비교
- 환각이 발생한 입력 오디오를 보존하여 재현 테스트

## MT 문제

### 속도

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 동시 접근 오류 (Already borrowed) | **확인됨** | 백엔드 로그에 직접 오류 메시지 기록됨 |
| max_length=512 고정 | **확인됨** | `nmt_service.py` 코드에서 확인 |
| 모델 크기 (NLLB 1.3B) | **확인됨** | LATENCY 로그에서 NMT 구간 수치 측정 가능 |

### 품질

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 단편 발화의 맥락 부재 | **미확인** | 발화가 짧을수록 앞뒤 문맥 없이 번역됨, 샘플 수집 필요 |
| 전문 용어 처리 (강의 도메인) | **미확인** | NLLB는 일반 도메인 학습 모델, 도메인 특화 평가 필요 |
| 다의어 해석 오류 | **미확인** | 실제 번역 결과 샘플 수집 및 유형 분류 필요 |
| 문법 오류 | **미확인** | 실제 번역 결과 샘플 수집 및 유형 분류 필요 |

### 미확인 항목 검증 방법

- 실제 번역 결과 텍스트를 20~30건 이상 수집
- 오류 유형별로 분류 (다의어 / 전문 용어 / 문법 / 맥락 부재)
- 짧은 발화(5단어 미만)와 긴 발화(10단어 이상) 번역 품질 비교

## 현재 구조의 속도 문제 요약

| 문제 | 위치 | 확인 여부 |
|------|------|----------|
| ASR → NMT → TTS 완전 직렬 처리 | `ws.py` process_audio | **✅ 개선됨** — TTS 제거로 ASR→NMT 완료 즉시 전송 |
| GPU 동시 접근 경합 (44s~253s 극단 지연) | `ws.py` 전역 싱글톤 | **✅ 적용 완료** — `_asr_semaphore` + `_nmt_semaphore` 분리 |
| asyncio.to_thread의 GPU 작업 비효율 | `ws.py` | **확인됨** |
| NMT max_length 고정값 | `nmt_service.py` | **확인됨** |
| TTS 전체 합성 후 일괄 전송 (스트리밍 미지원) | (제거됨) | **✅ 해결됨** — 백엔드 TTS 완전 제거, 클라이언트 WASM으로 이전 |
| ASR 결과를 NMT+TTS 완료 전까지 전송 안 함 | `ws.py` | **✅ 해결됨** — NMT 완료 즉시 텍스트 전송 (TTS 대기 제거) |

## 병렬 처리 + 순서 보장 설계 검토

### 아이디어

발화 1이 TTS까지 완전히 끝나지 않아도, 발화 2를 동시에 처리 시작하여 속도를 개선하는 방식.  
단, 최종 전송 순서는 반드시 발화 1 → 발화 2 순서를 보장해야 한다.

### 가능한 경우 — GPU 경합이 없을 때

발화 1이 **NMT 구간(CPU)** 에 진입한 시점에 발화 2가 들어오면 ASR GPU는 비어 있다.  
NMT(Helsinki)는 CPU 전용이므로 이 구간에서 발화 2가 GPU(ASR)를 자유롭게 사용할 수 있다.

```
발화1: ASR(GPU) → NMT(CPU) → 전송
발화2:              ASR(GPU) → NMT(CPU) → 전송
                   ↑ 발화1이 NMT 중일 때 GPU 유휴 → 발화2 ASR 즉시 시작 가능
```

TTS가 제거됨으로써 파이프라인이 단축되어 전송 지연이 추가로 감소합니다.

### 위험한 경우 — ASR GPU 동시 점유

NMT는 CPU로 이동했으므로 GPU 경합은 ASR 구간에서만 발생한다.  
`_asr_semaphore`(Semaphore(1))가 ASR GPU를 직렬화하여 동시 점유를 차단한다.

```
발화1: ASR(GPU) ←── _asr_semaphore 보유
발화2:     대기 중 ←── _asr_semaphore 대기 (발화1 ASR 완료 시 즉시 시작)
```

faster-whisper large-v3-turbo-korean은 약 3.1GB를 점유하므로  
8GB VRAM 환경에서 단일 ASR 직렬화 방식은 OOM 없이 안정적으로 동작한다.

### 구현 조건 (현재 적용 상태)

| 요소 | 구현 내용 |
|------|----------|
| seq_id (시퀀스 번호) | `_utterance_seq` 전역 카운터, 발화마다 증가 |
| 순서 보장 | seq 번호를 transcription 메시지에 포함해 프론트엔드에서 처리 |
| 발화2 ASR 시작 조건 | 발화1 ASR 완료 시 `_asr_semaphore` 해제 → 발화2 즉시 GPU 사용 |
| 과부하 방지 | `_MAX_QUEUED_AUDIO=2` 초과 또는 6s 이상 대기 시 자동 폐기 |

### 적용 결과

`ws.py` 수정 완료. `_asr_semaphore` + `_nmt_semaphore` 분리 구조로 구현되었으며,  
발화1 ASR 완료 즉시 발화2 ASR을 시작할 수 있어 파이프라인 병렬 처리가 실현되었다.

## 다음 주 액션

### 담당자별 수집 과제

| 담당 | 수집 항목 |
|------|----------|
| ASR 담당 | 발화 길이별 ASR 처리 시간 로그 30건, 환각 발생 샘플 오디오 보존 |
| MT 담당 | 번역 결과 30건 수집 후 오류 유형 분류 (다의어 / 전문 용어 / 문법 / 맥락) |

### 개선 우선순위 (원인 분석 후 적용 순서)

1. ✅ **GPU 경합 해결 — 세마포어 분리 완료** (`ws.py`, `_asr_semaphore` + `_nmt_semaphore` 분리, TTS 세마포어 밖)
2. ✅ **NMT CPU 이동 완료** — `NMT_ASR_DEVICE=cpu`, GPU 자원 경쟁 해소
3. ✅ **NMT 모델 교체 완료** — NLLB-1.3B → `Helsinki-NLP/opus-mt-ko-en`, CPU에서 0.1~0.2s
4. ✅ **ASR 모델 교체 완료** — Cohere → `ghost613/faster-whisper-large-v3-turbo-korean` (float16, CTranslate2)
5. ✅ **VAD 단독 전송 완료** — 1초 interval 제거, 발화 단위 정확 전송
6. ✅ **백엔드 TTS 완전 제거** — `tts_service.py` 미사용, `main.py`·`mode.py`·`ws.py`에서 모든 TTS 코드 제거. 클라이언트 WASM(piper-tts-web)으로 이전, NMT 완료 즉시 전송, 서버 부하 제거
7. ✅ **NMT max_length 동적 조정 완료** — `nmt_service.py`, `max(20, int(len(tokens) * 2.5))` 적용, 긴 발화 잘림 방지
8. 빠른 연속 발화 VAD 분할 — 현재 한 덩어리로 묶이는 문제 미해결

```
개선 전:  GPU 경합 → 44s~253s
1~5단계 완료 후: 짧고 명확한 발화 0.64~1.00s ✓ / 빠른 연속 발화(한 덩어리) 3.18s ✗
6단계(TTS 제거) 완료 후: TTS 합성 대기(0.5~2s) 추가 제거 → 파이프라인 지연 단축
```

> 속도 개선 전에 원인이 확인되지 않은 항목은 수집한 데이터를 바탕으로 재평가한다.

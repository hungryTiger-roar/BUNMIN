# ASR → MT → TTS 파이프라인 원인 분석

> 작성일: 2026-04-24 / 최종 수정: 2026-04-25  
> 목적: 파이프라인 개선 전 각 모델별 문제 원인을 확인된 것과 미확인된 것으로 분리하여 정리

---

## 분석 배경

현재 ASR → MT → TTS 파이프라인에서 속도와 품질 두 측면에서 문제가 발생하고 있다.  
한 사람이 전체 파이프라인을 수정하는 방식은 원인 파악과 효과 검증이 어렵기 때문에,  
다음 주부터 모델별 담당자를 지정하여 각자 원인을 깊이 파고드는 방식으로 전환한다.

---

## ASR 문제

### 속도

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 모델 자체 처리 시간 (Cohere 모델 크기) | **확인됨** | LATENCY 로그에서 ASR 구간 수치 측정 가능 |
| GPU 경합 (NMT와 GPU 동시 점유) | **✅ 적용 완료** | `asyncio.Semaphore(1)` 적용 — ASR·NMT 구간 직렬화, TTS는 세마포어 밖 분리 |
| 입력 오디오 길이와 처리 시간 비례 여부 | **미확인** | 짧은 발화(1~3s) vs 긴 발화(5s+) 구간별 비교 필요 |

### 품질 (환각)

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 노이즈 입력 (VAD 통과한 소음) | **확인됨** | 반복 문자, 중국어 혼입 패턴이 로그에 기록됨 |
| 짧고 조용한 발화 통과 | **확인됨** | `_validate_audio` 필터가 잡는 케이스 존재 |
| 모델 특성 (Cohere 한국어 환각 경향) | **미확인** | 동일 오디오로 다른 모델(Moonshine 등) 비교 실험 필요 |

### 미확인 항목 검증 방법

- LATENCY 로그를 발화 길이별(짧음/긺)로 분류하여 수십 건 수집
- 동일 오디오 샘플로 Cohere vs Moonshine v2 Korean 결과 비교
- 환각이 발생한 입력 오디오를 보존하여 재현 테스트

## MT 문제

### 속도

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 동시 접근 오류 (Already borrowed) | **확인됨** | 백엔드 로그에 직접 오류 메시지 기록됨 |
| max_length=512 고정 | **확인됨** | `nmt_service.py` 코드에서 확인 |
| 모델 크기 (NLLB 1.3B) | **확인됨** | LATENCY 로그에서 NMT 구간 수치 측정 가능 |

### 품질

| 원인 | 확인 여부 | 근거 |
|------|----------|------|
| 단편 발화의 맥락 부재 | **미확인** | 발화가 짧을수록 앞뒤 문맥 없이 번역됨, 샘플 수집 필요 |
| 전문 용어 처리 (강의 도메인) | **미확인** | NLLB는 일반 도메인 학습 모델, 도메인 특화 평가 필요 |
| 다의어 해석 오류 | **미확인** | 실제 번역 결과 샘플 수집 및 유형 분류 필요 |
| 문법 오류 | **미확인** | 실제 번역 결과 샘플 수집 및 유형 분류 필요 |

### 미확인 항목 검증 방법

- 실제 번역 결과 텍스트를 20~30건 이상 수집
- 오류 유형별로 분류 (다의어 / 전문 용어 / 문법 / 맥락 부재)
- 짧은 발화(5단어 미만)와 긴 발화(10단어 이상) 번역 품질 비교

## 현재 구조의 속도 문제 요약

| 문제 | 위치 | 확인 여부 |
|------|------|----------|
| ASR → NMT → TTS 완전 직렬 처리 | `ws.py` process_audio | **확인됨** |
| GPU 동시 접근 경합 (44s~253s 극단 지연) | `ws.py` 전역 싱글톤 | **✅ 적용 완료** — 세마포어로 ASR·NMT 직렬화 |
| asyncio.to_thread의 GPU 작업 비효율 | `ws.py` | **확인됨** |
| NMT max_length 고정값 | `nmt_service.py` | **확인됨** |
| TTS 전체 합성 후 일괄 전송 (스트리밍 미지원) | `tts_service.py` | **확인됨** |
| ASR 결과를 NMT+TTS 완료 전까지 전송 안 함 | `ws.py` | **확인됨** |

## 병렬 처리 + 순서 보장 설계 검토

### 아이디어

발화 1이 TTS까지 완전히 끝나지 않아도, 발화 2를 동시에 처리 시작하여 속도를 개선하는 방식.  
단, 최종 전송 순서는 반드시 발화 1 → 발화 2 순서를 보장해야 한다.

### 가능한 경우 — GPU 경합이 없을 때

발화 1이 **TTS 구간(CPU)** 에 진입한 시점에 발화 2가 들어오면 GPU는 비어 있다.  
Piper TTS는 CPU 전용이므로 이 구간에서 발화 2가 GPU(ASR, NMT)를 자유롭게 사용할 수 있다.

```
발화1: ASR(GPU) → NMT(GPU) → TTS(CPU) ──────────→ 버퍼 대기 → 전송
발화2:                          ASR(GPU) → NMT(GPU) → TTS(CPU) → 버퍼 대기 → 전송
                                ↑ 발화1이 TTS 중일 때 GPU 유휴 → 병렬 처리 가능
```

발화 2가 먼저 완료되더라도 발화 1이 전송될 때까지 버퍼에 대기시켜 순서를 보장한다.

### 위험한 경우 — GPU 동시 점유

발화 1이 **ASR 또는 NMT 구간(GPU)** 에 있는데 발화 2도 GPU를 동시에 사용하면,  
8GB VRAM 환경에서 Cohere ASR + NLLB-1.3B가 이미 5~6GB를 점유하고 있어 OOM 위험이 있다.

```
발화1: ASR(GPU) ─────────────
발화2:     ASR(GPU) ←── VRAM 충돌 위험
```

### 구현 조건

| 필요 요소 | 설명 |
|----------|------|
| seq_id (시퀀스 번호) | 각 발화에 순서 번호 부여 |
| ordered output 버퍼 | 완료된 결과를 dict에 보관, seq_id 순서대로 전송 |
| 파이프라인 구간 상태 추적 | 발화 1이 TTS에 진입한 시점에만 발화 2 처리 시작 허용 |

### 적용 시점

현재 `ws.py` 구조를 상당 부분 변경해야 하는 작업이다.  
원인 분석이 완료되고 ASR / MT 담당자가 실제 처리 시간 데이터를 확보한 이후에 설계한다.

## 다음 주 액션

### 담당자별 수집 과제

| 담당 | 수집 항목 |
|------|----------|
| ASR 담당 | 발화 길이별 ASR 처리 시간 로그 30건, 환각 발생 샘플 오디오 보존 |
| MT 담당 | 번역 결과 30건 수집 후 오류 유형 분류 (다의어 / 전문 용어 / 문법 / 맥락) |

### 개선 우선순위 (원인 분석 후 적용 순서)

1. ✅ **GPU 경합 해결 — 세마포어 적용 완료** (`ws.py`, ASR·NMT 직렬화 + TTS 세마포어 밖 분리)
2. NMT max_length 동적 조정 — `nmt_service.py` 1줄 변경, 즉각 효과
3. TTS 구간에서 다음 발화 GPU 선처리 허용 — 세마포어 안정화 후 적용
4. ASR 모델 교체 검토 — Moonshine v2 Korean (파이프라인 안정화 후)

```
현재:  경합 발생 → 44s~253s
1단계: 세마포어 ✅ → 순차 처리, 경합 제거 (목표: 안정적 2~4s)
2단계: max_length 동적 조정 → NMT 구간 단축
3단계: TTS 구간 병렬 허용 → 체감 속도 개선
4단계: ASR 모델 교체 → GPU 점유 시간 자체 감소
```

> 속도 개선 전에 원인이 확인되지 않은 항목은 수집한 데이터를 바탕으로 재평가한다.

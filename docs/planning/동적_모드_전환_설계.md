# 동적 모델 모드 전환 설계

## 배경
- VLM(슬라이드 번역)과 실시간 모델(ASR/NMT/OCR)을 동시에 로드하면 GPU 메모리 부족
- 슬라이드 번역 후 강의 시작 시 모델 전환 필요

---

## 상태 다이어그램

> 실제 구현은 IDLE / SLIDE / REALTIME 3가지 모드로 단순화되어 있습니다.
> 아래 다이어그램은 각 모드에서 일어나는 동작 흐름을 나타냅니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                        서버 모드 다이어그램                        │
└─────────────────────────────────────────────────────────────────┘

                         ┌──────────┐
                         │  IDLE    │
                         │ (시작)   │
                         └────┬─────┘
                              │
                ┌─────────────┴──────────────┐
                │                            │
                │ SKIP_STARTUP_MODELS=false   │ SKIP_STARTUP_MODELS=true
                │ (기본값)                    │
                ▼                            ▼
     ┌──────────────────────┐    ┌──────────────────────┐
     │      REALTIME        │    │        SLIDE         │
     │  ASR/NMT/OCR 로드됨  │    │  VLM 온디맨드 대기    │
     └──────────┬───────────┘    └──────────┬───────────┘
                │                           │
                │ POST /api/mode/slide       │ POST /api/mode/realtime
                │ (실시간 모델 언로드)         │ (VLM 언로드 → ASR/NMT 로드)
                ▼                           ▼
     ┌──────────────────────┐    ┌──────────────────────┐
     │        SLIDE         │    │      REALTIME        │
     │  VLM 온디맨드 대기    │    │  ASR/NMT 로드됨      │
     └──────────────────────┘    └──────────────────────┘
```

> 주의: 최초 시작 시 REALTIME 모드는 ASR/NMT/OCR 3종을 모두 로드하지만,
> SLIDE → REALTIME 전환은 OCR을 재로드하지 않습니다 (OCR은 슬라이드 번역 전용).

---

## 모드 전환 시점 옵션 비교

### 옵션 A: 강의 시작 버튼 클릭 시 전환

```
번역 완료 ──► [강의 시작 클릭] ──► 로딩... ──► 강의 시작
                                  (10~30초)
```

**장점:**
- 구현 단순
- 불필요한 전환 없음 (번역만 하고 끝낼 수 있음)

**단점:**
- 강의 시작까지 대기 시간 발생 (10~30초)
- UX 불편

---

### 옵션 B: 번역 완료 후 백그라운드 전환 (미리 준비)

```
번역 완료 ──► 백그라운드 전환 시작 ──► [강의 시작 클릭] ──► 즉시 시작
              (VLM 언로드, 실시간 로드)      (이미 준비됨)
```

**장점:**
- 강의 시작 즉시 가능
- UX 좋음

**단점:**
- 다시 번역하려면 VLM 재로드 필요
- 번역만 하고 끝내면 불필요한 전환 발생

---

### 옵션 C: 번역 완료 후 일정 시간 대기 후 전환

```
번역 완료 ──► 30초 대기 ──► 추가 번역 없으면 ──► 백그라운드 전환
                │
                └──► 추가 번역 있으면 ──► 타이머 리셋
```

**장점:**
- 추가 번역 가능성 고려
- 자연스러운 전환

**단점:**
- 구현 복잡
- 타이밍 애매할 수 있음

---

### 옵션 D: 사용자 선택 (준비 버튼)

```
번역 완료 ──► [강의 준비] 버튼 표시
              │
              ├──► 클릭하면 ──► 모드 전환 ──► [강의 시작] 활성화
              │
              └──► 다른 슬라이드 업로드 ──► 버튼 숨김
```

**장점:**
- 사용자 의도 명확
- 불필요한 전환 없음

**단점:**
- 버튼 하나 더 필요
- 클릭 한 번 더

---

## 검토 필요 사항

1. **모드 전환 소요 시간**: 실제로 몇 초 걸리는지 측정 필요
2. **사용자 행동 패턴**: 번역 후 바로 강의 시작하는 비율?
3. **GPU 메모리**: 두 모델 동시 로드 가능한지? (불가능하면 전환 필수)

---

## API 엔드포인트

```
GET  /api/mode/current   - 현재 모드 확인
POST /api/mode/slide     - 슬라이드 모드 (실시간 모델 ASR/NMT/OCR 언로드)
POST /api/mode/realtime  - 실시간 모드 (VLM 언로드 → ASR/NMT-ASR 로드)
```

        ▼

### 재번역 시나리오

```
  [REALTIME_READY 상태에서 슬라이드 재업로드]
        │
        ▼
  ┌───────────────────────────────────────┐
  │  프론트엔드: POST /api/mode/slide     │
  │  (실시간 모델 언로드)                  │
  └───────────────┬───────────────────────┘
                  │
                  ▼
  ┌───────────────┐
  │  VLM 재로드   │  ← 시간 소요 (10~30초)
  │  번역 진행    │
  └───────────────┘
```

### 프론트엔드 구현 포인트

1. **번역 완료 감지**: 슬라이드 번역 상태가 `completed`가 되면
2. **모드 전환 호출**: `POST /api/mode/realtime` 비동기 호출
3. **UI 상태 표시**: "강의 준비 중..." 로딩 표시 (선택)
4. **강의 시작 버튼**: 모드 전환 완료 후 활성화

### 백엔드 구현 완료

- `POST /api/mode/realtime`: VLM 언로드 → ASR/NMT-ASR 로드 (OCR은 슬라이드 전용이라 재로드 안 함)
- `POST /api/mode/slide`: 실시간 모델 ASR/NMT/OCR 언로드 (VLM은 온디맨드)
- `GET /api/mode/current`: 현재 모드 확인

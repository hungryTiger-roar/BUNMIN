# 프론트엔드 설계

## 개요

React + TypeScript 기반 웹앱.
Electron 데스크탑 앱으로도 패키징 가능하며, 강의자와 수강자 모두 동일한 앱을 사용하고 역할만 다릅니다.

---

## 기술 스택

| 분류 | 기술 |
|------|------|
| Framework | React + TypeScript |
| 번들러 | Vite |
| 데스크탑 (선택) | Electron (루트 package.json에서 관리) |
| 상태 관리 | Zustand |
| 스타일링 | Tailwind CSS |
| 실시간 통신 | WebSocket (native) |
| WASM TTS | piper-tts-web (CPU ONNX, 메인 스레드) |

## 화면 구성

### 1. 홈 (역할 선택)
- 강의자로 시작 / 수강자로 참여 선택

### 2. 참여 (수강자 이름 입력) — `/` (base URL)
- 강의자 측 공유 링크는 **`http://<lan_ip>:<port>/`** 로 발급 (Start 컴포넌트가 마운트되어 이름 입력 화면 노출).
- 이름 입력 후 강의 참여
- 이름 로컬스토리지 저장 옵션
- 한국어 / English UI 전환
- **"강의 참여" 버튼 클릭 시 navigate state(`autoEnter: true`)를 Student 화면에 전달 → 클릭의 transient activation을 활용해 AudioContext 즉시 언락**

### 3. 로딩
- 백엔드 AI 모델 로딩 진행률 표시
- 모델별 로딩 상태 (ASR / NMT-ASR / OCR / VLM)

### 4. 강의자 화면
- **강의자료 모드**: PDF 업로드 → 슬라이드 뷰어로 페이지 제어
- **화면공유 모드**: 화면 캡처 후 수강자에게 실시간 전송
- 마이크 ON/OFF → 음성 실시간 번역
- 강의 시작 / 일시정지 / 종료 제어
- 실시간 자막 미리보기
- 마우스 커서 위치 수강자에게 실시간 공유 (상대 좌표 0~1)

### 5. 수강자 화면
- 슬라이드 또는 화면공유 수신 표시
- **음성 출력 모드 선택** (`audioLang`, 단일 선택):
  - `Off` — 무음, 자막만
  - `원본 (Original)` — 강사 마이크 음성을 WebRTC 로 받아 DelayNode (적응형, 초기 10s, 2~20s 범위) 경유로 재생
  - `영어 (English)` — 번역 텍스트를 브라우저 내 piper-tts-web 으로 합성 (en_US-lessac-medium)
  - `독일어 / 스페인어 / 러시아어` — 각 piper voice (다국어 TTS 정책 보류 — 현재 영어만 실재생, 나머지는 dropdown 유지)
  - `원본 ↔ 영어` 토글 시 instant 전환 (TTS 엔진은 항시 합성 + GainNode mute/unmute, silence gap 없음)
  - Start 페이지 경유 시 즉시 언락 / 직접 접속 시 첫 인터랙션에서 자동 언락
  - 볼륨 슬라이더 · 음소거 제어
- **자막 표시**: 메인 자막 + 부자막 동시 표시, 언어 독립 선택
- 자막 커스터마이징: 글자 크기 · 스타일(기본/테두리/글로우) · 위치(상단/하단)
- 화면 비율 선택: 16:9 / 4:3 / 5:3
- 라이트 / 다크 / 그라데이션 테마 3-모드 토글
- 채팅 (강의자 ↔ 수강자)
- 참여자 목록 (강의자 · 수강자)
- 강의 자료 다운로드 (원본 PDF / 번역 PDF)
- 강의 종료 후 자막 다운로드 모달 (TXT / SRT)
- 강의자 커서 스팟라이트 오버레이 (ref 기반 DOM 직접 조작)

## 수강자 음성 출력 — 이중 파이프라인

수강자측 음성 출력은 **두 개의 독립 Web Audio 파이프라인**으로 구성됩니다. 사용자가 `audioLang` 으로 선택하면 둘 중 하나만 들리고 (mutually exclusive), 토글 시점에 GainNode mute/unmute 로 즉시 전환됩니다.

| 파이프라인 | 신호 경로 | 트리거 |
|---|---|---|
| **원본 음성 (WebRTC)** | 강사 마이크 → WebRTC → MediaStreamSource → DelayNode (적응형 `currentDelay`, 2~20s) → GainNode → destination | `audioLang === 'original'` |
| **WASM TTS** | 백엔드 transcription → useDelayBufferPlayer 스케줄 → piper-tts-web 합성 → AudioBufferSource → GainNode → destination | `audioLang === 'en'` (de/es/ru 는 dropdown 만 유지) |

두 파이프라인은 별도 AudioContext 를 사용하며 (TTS context, Original audio context), 시스템 audio 출력에서 합쳐집니다.

### 옵션 1: TTS 항시 합성 정책

`audioLang` 이 무엇이든 영어 TTS 는 **항상 합성·재생** 됩니다. 다만 `audioLang !== 'en'` 이면 TTS 의 master GainNode 가 0 으로 mute. 토글 시 instant 전환 (silence gap 없음) 을 위해 의도된 trade-off (CPU ~3% 추가).

### Wall-clock + delay-buffer sync 모델

`useDelayBufferPlayer` 가 강사 wall-clock 기준 **적응형 lag (`currentDelay`)** 으로 모든 학생 출력을 정렬합니다.

```
강사 wall T_L 사건
   ├─ 원본 음성: WebRTC 송출 → DelayNode 가 currentDelay 만큼 지연 후 출력
   ├─ 그림/포인트/페이지: scheduleAt(lecTs + offset + currentDelay, apply) 로 setTimeout
   ├─ 자막(원본 모드): scheduleAt 시점 즉시 commitSubtitle (wall-clock anchor)
   └─ TTS(영어 모드): scheduleAt 시점에 audio queue 진입 → audioStartedAt 에 자막 commit (TTS anchor)
```

- `offset` = 학생 wall − 강사 wall (EWMA 추정, visual event 의 lecturerTimestamp 기반)
- 모든 경로가 같은 `lecturer_wall + offset + currentDelay` 로 수렴 → audio/visual/subtitle 동기.
- **`currentDelay` 적응형 계산** (`useDelayBufferPlayer`): `recentP90(처리시간) + max(300ms, stddev × 1.5)`
  - 늘릴 땐 즉시 (desync 방지 우선), 줄일 땐 천천히 EWMA(α=0.25) — 원본 음성 jump 완화
  - silence 구간에서 큐가 비면 자연히 최솟값으로 수렴
- **범위**: `DELAY_MIN_MS=2000`, `DELAY_MAX_MS=20000` (2초 ~ 20초)
- **초기값**: `VITE_SYNC_DELAY_MS` 환경변수 (기본 10000ms = 10초). 첫 발화 전까진 이 값 사용, 이후 실측 기반 동적 조정.
- 자세한 환경 설정은 [환경설정](../setting/환경설정.md) 참고.

### 원본 음성 (WebRTC) 파이프라인 상세

```
강사 Lecturer.tsx
    │ useAudioCapture.micStream
    ▼
RTCPeerConnection (학생당 1개)
    │ addTrack(audioTrack, micStream)
    │ createOffer / setLocalDescription → send via WebSocket
    ▼
WebSocket signaling (offer / answer / ICE)
    ▼
학생 Student.tsx (handleWebRtcOffer)
    │ ontrack(kind='audio')
    │   ├─ <audio> srcObject = new MediaStream([track]) + muted + play()  ← keepalive
    │   └─ createMediaStreamSource → connect(delayNode)
    ▼
DelayNode (적응형 currentDelay, 2~20s) → GainNode → ctx.destination
```

- `<audio>` element 는 **트랙 keepalive** 용 (muted, display:none) — Chrome 이 element 가 "playing" 상태일 때만 WebRTC 디코더가 MediaStreamSource 로 frame 을 흘려보내는 동작 보호.
- `ontrack` 시 매번 `srcObject` 재할당 + `play()` 명시 호출 → 강사 mic/screen 토글로 인한 재협상 후에도 디코더 활성 유지.
- GainNode 변화는 10ms 선형 ramp — sample-boundary 클릭 방지 + 토글 즉시 느낌.
- **iceServers**: Google STUN + 로컬 TURN (`turn:<lan_ip>:47878`, UDP+TCP 두 transport — 표준 3478 충돌 회피). LAN IP 는 `getLanIp()` (`@/lib/network`) 가 `/network/info` 응답을 캐시해 공급. 강사 노트북이 곧 TURN host — `turn-server/index.js` (node-turn) 가 `npm run dev` 로 자동 실행, Electron 운영판은 `main.cjs` 의 `startTurnServer()` 가 기동. SSAFY 같은 P2P 차단 환경에서도 client→server relay 로 우회.
- **Opus DTX 비활성화**: `createOffer` 후 SDP 의 `usedtx=1` 제거. 묵음 구간에도 packet stream 유지 → 발화 재개 시 packet 끊김 → jitter buffer underrun → "지지직" 아티팩트 방지. 대역폭 +5kbps/stream (무시 가능).

### 백그라운드 복귀 시 sync 보존 (옵션 B)

모바일/iOS 등에서 학생이 백그라운드로 이동하면 AudioContext 가 suspend 되고 DelayNode buffer 가 freeze 됩니다. 단순 `ctx.resume()` 만 하면 stale buffer 가 출력되어 visual/자막과 N초 desync 됩니다.

`visibilitychange` / `pageshow` / `focus` 시점에:
1. ctx 가 suspended 상태인지 확인 → resume
2. 기존 DelayNode + MediaStreamSource 폐기 → 새로 생성
3. WebRTC stream 재연결, 새 DelayNode 가 빈 buffer 로 시작

복귀 후 현재 `currentDelay` 만큼 무음 (새 buffer 채우는 시간) 이후 sync 유지된 채 정상 복구. silence 후 복귀면 currentDelay 가 작아 짧고, 활발한 발화 도중 복귀면 길어짐. 데스크톱 브라우저는 보통 백그라운드에서도 ctx 가 running 이라 이 경로 미발화.

### WASM TTS (piper-tts-web)

수강자 브라우저 내에서 CPU ONNX 로 음성을 합성합니다 — 서버에서 음성 파일을 전송하지 않습니다.

#### 지원 언어 및 음성 모델

| audioLang | Piper Voice ID | 비고 |
|-----------|---------------|------|
| `en` | en_US-lessac-medium | 영어 (기본) — 항시 합성, GainNode 로 mute 제어 |
| `de` | de_DE-thorsten-medium | 독일어 (현재 정책상 합성 안 함, dropdown 유지) |
| `es` | es_MX-ald-medium | 스페인어 (동일) |
| `ru` | ru_RU-irina-medium | 러시아어 (동일) |
| `original` / `ko` / `off` | — | TTS 합성은 진행되지만 GainNode 가 mute. 'original' 은 별도 WebRTC 파이프라인이 출력 |

#### 아키텍처

```
useWebSocket (onTranscription 콜백)
    │ { text, speechStartAt, sentAt, commitSubtitle }
    ▼
useDelayBufferPlayer.enqueueSentence
    │ scheduleAt(speechStartAt + offset + currentDelay, ...)
    ▼
audioQueueRef (FIFO 직렬 큐)
    │ 이전 sentence 의 ended Promise await
    ▼
useTTS.playSentence(text, 'en')   ← 항시 'en'
    │
    ├── PiperWebEngine.generate()   (메인 스레드, 내부 Worker spawn)
    ├── AudioContext.decodeAudioData()
    ▼
AudioBufferSource → sourceGain → masterGain → destination
                                   (audioLang!=='en' 이면 0)
```

#### 메모리 관리 (OOM 방지)

- TTS 엔진은 `'en'` 고정 — 언어 토글 시 엔진 재초기화 없음.
- Heap 사용률 80% 초과 시 엔진 재생성 (5분 cooldown, 발화 중엔 보류 후 idle 진입 시점에 재생성).
- 재생 중 task 의 source/gain 노드는 onended 에서 disconnect → graph 누수 방지.

### 오디오 언락 정책

브라우저 autoplay 정책으로 인해 두 AudioContext 모두 사용자 제스처 후에만 활성화됩니다.

| 접근 경로 | 언락 시점 |
|-----------|----------|
| Start 페이지 "강의 참여" 클릭 경유 | Student.tsx 진입 직후 즉시 언락 (transient activation 활용) |
| 직접 URL 접속 / 새로고침 | 화면의 첫 클릭 · 키 입력 · 터치 시 자동 언락 (`{ once: true }` 리스너) |

언락 함수 (`unlockAudio`) 가 두 ctx (TTS + Original) 를 동시에 초기화.

### WASM 런타임 파일

piper-tts-web 이 런타임에 fetch 하는 파일들은 `public/` 하위에 정적 배치해야 합니다.
크기가 크므로 git 에서 제외(`gitignore`)하며, `node_modules/piper-tts-web/dist/` 에서 수동으로 복사합니다.

| 디렉터리 | 내용 | 크기 |
|----------|------|------|
| `public/onnx/` | ONNX Runtime WASM | ~32MB |
| `public/piper/` | Piper 음소화 엔진 WASM + 데이터 | ~18MB |
| `public/worker/` | Piper 내부 Web Worker JS | ~60MB |

## URL 처리 (Electron / 웹 자동 감지)

`src/lib/api.ts`에서 환경을 자동 감지합니다:

```typescript
// Electron 프로덕션: file://로 로드되므로 백엔드 주소를 127.0.0.1:48000으로 고정
// ('localhost'는 Windows에서 IPv6(::1)로 풀려 uvicorn IPv4 전용 서버에 닿지 않을 수 있음)
// Electron dev 모드는 Vite dev 서버에서 로드되므로 프록시 경유
export const API_BASE = isElectron && !isViteDev
  ? 'http://127.0.0.1:48000' // Electron 프로덕션: 직접 백엔드 호출
  : ''                      // 그 외: Vite 프록시(/api, /slides, /health 등) 경유

export const WS_BASE = isElectron && !isViteDev
  ? 'ws://127.0.0.1:48000'
  : `${window.location.protocol === 'https:' ? 'wss:' : 'ws:'}//${window.location.host}`

export const WS_PIPELINE_URL = `${WS_BASE}/ws/pipeline`
```

웹 개발 환경에서 백엔드 주소는 `BACKEND_URL` 환경변수로 제어합니다 (기본값: `http://localhost:48000`). 자세한 내용은 [환경설정](../setting/환경설정.md)을 참고하세요.

# 브랜치 전략

## 구조

```
master
└── dev
    ├── S14P31S205-{이슈번호}-{영어설명}
    ├── S14P31S205-{이슈번호}-{영어설명}
    └── S14P31S205-{이슈번호}-{영어설명}

master (긴급)
└── hotfix/S14P31S205-{이슈번호}-{영어설명}  ← master 긴급 수정
```

## 네이밍 규칙

- 형식: `S14P31S205-{이슈번호}-{영어설명}`
- 설명은 영어로 작성
- 띄어쓰기 대신 `-` 사용
- 기능 추가, 버그 수정, 디자인, 리팩토링 등 모든 작업에 동일 형식 적용

### 예시
```
S14P31S205-80-add-login-feature
S14P31S205-81-fix-audio-bug
S14P31S205-82-update-slide-design
S14P31S205-83-refactor-nmt-service
```

## hotfix 규칙

- **운영(master) 긴급 수정**: `master`에서 직접 분기 → 수정 후 `master`와 `dev` 양쪽에 머지
- **개발(dev) 긴급 수정**: `dev`에서 직접 분기 → 수정 후 `dev`에 머지

## 머지 흐름

```
S14P31S205-{번호}-{내용} → dev → master
hotfix/* (dev) → dev
hotfix/* (master) → master → dev
```

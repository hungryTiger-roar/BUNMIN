# 수강자 N명 환경에서의 네트워크 대역폭 개선안

> 작성일: 2026-04-25  
> 목적: TTS 오디오 전송 시 수강자 수 증가에 따른 서버 네트워크 부하 문제 해결  
> **✅ 2026-04-29: TTS 클라이언트 WASM 이전으로 근본 해결 — 하단 참고**

---

## 배경 — 왜 문제가 되는가

강의자가 한 문장을 말하면 서버는 다음 순서로 처리합니다.

```
강의자 발화 → ASR(한국어 인식) → MT(영어 번역) → TTS(음성 합성) → 수강자 전송
```

이때 TTS로 만든 **음성 파일을 수강자 한 명 한 명에게 개별로 복사해서 전송**합니다.

### 비유로 이해하기

> 학교에서 선생님이 인쇄물을 나눠줄 때,  
> 복사기에서 **학생 수만큼 출력해서** 한 명씩 건네는 방식입니다.  
> 학생이 10명이면 10장, 100명이면 100장을 복사해서 전달합니다.  
> 당연히 학생이 많을수록 시간과 비용이 선형으로 늘어납니다.

현재 구조에서는 발화 1건당 전송량이 아래와 같습니다.

| 수강자 수 | 발화 1건 전송량 |
|----------|--------------|
| 10명     | 약 500KB     |
| 100명    | 약 5MB       |
| 1,000명  | 약 50MB      |

강의 중 발화는 분당 수십 건 발생하므로, 수강자가 많아질수록 서버의 네트워크 대역폭이 한계에 도달합니다.

---

## ✅ 근본 해결 — TTS 클라이언트 WASM 이전 (2026-04-29 적용)

백엔드 TTS 파이프라인을 완전히 제거하고, 수강자 브라우저 내 **piper-tts-web (CPU WASM)** 으로 이전했습니다.

```
[이전] 강의자 발화 → ASR → MT → TTS(서버) → 음성파일 × 수강자 수 전송
[이후] 강의자 발화 → ASR → MT → 번역 텍스트 1건 전송 → 수강자 브라우저 WASM TTS
```

| 수강자 수 | 이전 전송량 (WAV) | 이후 전송량 (텍스트) |
|----------|-----------------|-------------------|
| 10명     | 약 500KB | 약 1KB |
| 100명    | 약 5MB   | 약 1KB |
| 1,000명  | 약 50MB  | 약 1KB |

서버는 번역된 텍스트만 브로드캐스트하므로, 수강자 수와 무관하게 발화 1건당 전송량이 일정합니다.

**추가 효과**:
- 서버 CPU 부하 감소 (TTS 합성 연산 제거)
- 파이프라인 지연 감소 (NMT 완료 즉시 전송, TTS 대기 없음)
- 수강자별 언어 선택 가능 (en/de/es/ru — 각자 원하는 언어로 재생)

---

## (참고) 기존 검토 개선안

아래 내용은 서버 TTS 전송 구조 기준의 개선안입니다. 현재는 근본 해결로 불필요해졌습니다.

### 개선안 1 — 오디오가 필요 없는 수강자에게는 음성을 보내지 않는다

수강자가 오디오를 끄거나 켤 때, 그 상태를 서버에 알립니다.  
서버는 **오디오를 켠 수강자에게만** 음성 파일을 포함해서 전송합니다.

```
[오디오 ON 수강자]  → 자막 텍스트 + 음성 파일 수신
[오디오 OFF 수강자] → 자막 텍스트만 수신
```

### 개선안 2 — 음성 파일 크기를 압축한다

TTS가 WAV를 생성한 직후, 서버에서 **Opus 형식으로 변환**한 뒤 전송합니다.

```
기존: TTS → WAV(50KB) → 수강자 전송
개선: TTS → WAV → Opus 변환(5~10KB) → 수강자 전송
```

### 두 가지 개선을 함께 적용하면 (참고용)

| 상황 | 발화 1건 전송량 |
|------|--------------|
| 당시 현재 (100명, WAV) | 약 5MB |
| 1번만 적용 (20명 오디오 ON, WAV) | 약 1MB |
| 2번만 적용 (100명, Opus) | 약 500KB~1MB |
| 1번 + 2번 함께 (20명 오디오 ON, Opus) | 약 100KB~200KB |
| **✅ WASM 이전 (현재)** | **약 1KB (수강자 수 무관)** |

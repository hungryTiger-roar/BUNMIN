# 수강자 N명 환경에서의 네트워크 대역폭 개선안

> 작성일: 2026-04-25  
> 목적: TTS 오디오 전송 시 수강자 수 증가에 따른 서버 네트워크 부하 문제 해결

---

## 배경 — 왜 문제가 되는가

강의자가 한 문장을 말하면 서버는 다음 순서로 처리합니다.

```
강의자 발화 → ASR(한국어 인식) → MT(영어 번역) → TTS(음성 합성) → 수강자 전송
```

이때 TTS로 만든 **음성 파일을 수강자 한 명 한 명에게 개별로 복사해서 전송**합니다.

### 비유로 이해하기

> 학교에서 선생님이 인쇄물을 나눠줄 때,  
> 복사기에서 **학생 수만큼 출력해서** 한 명씩 건네는 방식입니다.  
> 학생이 10명이면 10장, 100명이면 100장을 복사해서 전달합니다.  
> 당연히 학생이 많을수록 시간과 비용이 선형으로 늘어납니다.

현재 구조에서는 발화 1건당 전송량이 아래와 같습니다.

| 수강자 수 | 발화 1건 전송량 |
|----------|--------------|
| 10명     | 약 500KB     |
| 100명    | 약 5MB       |
| 1,000명  | 약 50MB      |

강의 중 발화는 분당 수십 건 발생하므로, 수강자가 많아질수록 서버의 네트워크 대역폭이 한계에 도달합니다.

---

## 개선안 1 — 오디오가 필요 없는 수강자에게는 음성을 보내지 않는다

### 문제

현재 수강자가 앱에서 **오디오 끄기**를 눌러도, 서버는 이 사실을 모릅니다.  
서버는 모든 수강자에게 항상 음성 파일을 전송하고 있습니다.  
오디오를 끈 수강자는 받고도 재생하지 않는 데이터를 계속 수신합니다.

### 해결 방향

수강자가 오디오를 끄거나 켤 때, 그 상태를 서버에 알립니다.  
서버는 **오디오를 켠 수강자에게만** 음성 파일을 포함해서 전송합니다.

```
[오디오 ON 수강자]  → 자막 텍스트 + 음성 파일 수신
[오디오 OFF 수강자] → 자막 텍스트만 수신
```

### 기대 효과

예를 들어 수강자 100명 중 20명만 오디오를 켜고 있다면,  
음성 파일 전송량이 기존 대비 **80% 감소**합니다.

| 상황 | 발화 1건 전송량 |
|------|--------------|
| 개선 전 (100명 전원) | 약 5MB |
| 개선 후 (20명만 오디오 ON) | 약 1MB |

### 구현 범위

- **프론트엔드**: 오디오 ON/OFF 상태가 바뀔 때 서버에 메시지 전송
- **백엔드**: 수강자별 오디오 설정 상태 저장, 전송 시 조건 분기

---

## 개선안 2 — 음성 파일 크기를 압축한다

### 문제

현재 TTS(Piper)가 생성하는 음성 파일은 **WAV 형식**입니다.  
WAV는 압축하지 않은 원본 형태라 파일 크기가 큽니다.

### 비유로 이해하기

> 사진을 저장할 때 **BMP(무압축)** 와 **JPEG(압축)** 의 차이와 같습니다.  
> 같은 사진이라도 BMP는 5MB, JPEG는 200KB일 수 있습니다.  
> 품질 차이가 거의 없는데도 크기는 25배 차이가 납니다.  
>  
> 음성 파일도 마찬가지입니다.  
> WAV(무압축)를 Opus(압축)로 바꾸면 크기가 5~10배 줄어들고,  
> 실제 들리는 품질 차이는 거의 없습니다.

### 해결 방향

TTS가 WAV를 생성한 직후, 서버에서 **Opus 형식으로 변환**한 뒤 전송합니다.  
브라우저는 Opus 형식을 기본으로 지원하므로 프론트엔드 변경은 최소화됩니다.

```
기존: TTS → WAV(50KB) → 수강자 전송
개선: TTS → WAV → Opus 변환(5~10KB) → 수강자 전송
```

### 기대 효과

| 형식 | 발화 1건 파일 크기 | 100명 전송량 |
|------|----------------|------------|
| WAV (현재) | 약 50KB | 약 5MB |
| Opus (개선 후) | 약 5~10KB | 약 500KB~1MB |

### 구현 범위

- **백엔드**: TTS 합성 후 Opus 인코딩 단계 추가 (`ffmpeg` 또는 Python `opuslib` 활용)
- **프론트엔드**: 변경 없음 (브라우저 AudioContext가 Opus 자동 지원)

---

## 두 가지 개선을 함께 적용하면

| 상황 | 발화 1건 전송량 |
|------|--------------|
| 현재 (100명, WAV) | 약 5MB |
| 1번만 적용 (20명 오디오 ON, WAV) | 약 1MB |
| 2번만 적용 (100명, Opus) | 약 500KB~1MB |
| 1번 + 2번 함께 (20명 오디오 ON, Opus) | 약 100KB~200KB |

현재 대비 **최대 25~50배 전송량 감소**가 가능합니다.

---

## 적용 우선순위

| 순위 | 항목 | 이유 |
|------|------|------|
| 1 | 오디오 ON/OFF 연동 | 코드 변경이 단순하고 효과가 즉각적 |
| 2 | Opus 압축 | 인코딩 추가 필요하나 모든 수강자에게 기본 적용됨 |
